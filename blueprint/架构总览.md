# 文山州 AI 创作公共知识库系统 · 架构总览

> 将 D:\WenShanKB 从个人 Obsidian 知识库升级为可部署、可管理、可进化的公共知识库系统。

---

## 一、系统定位

**文山州 AI 创作公共知识库（WenShan Knowledge Hub）** 是一个面向文山州互联网协会会员、内容创作者和普通百姓的公共知识服务平台。核心价值：

| 维度 | 当前（Obsidian 知识库） | 升级后（公共知识库系统） |
|------|------------------------|------------------------|
| 访问方式 | 本地 Obsidian 客户端 | Web 浏览器 / API / Agent |
| 协作模式 | 单人编辑 | 多角色协同审核 |
| 内容更新 | 手动 | 自动采集 → AI 摘要 → 人工审核 |
| 知识检索 | wikilink 跳转 | 全文搜索 + 语义检索 + Agent 自动注入 |
| 质量保障 | 个人规范 | 自动化检查 + 用户纠错反馈 |
| 用户体系 | 无 | 四级角色 + 权限控制 |

### 2.1 展示层

| 组件 | 技术选型 | 职责 |
|------|----------|------|
| Web 前端 | Next.js 14 (App Router) + TailwindCSS + Shadcn/ui | 知识库浏览、搜索、管理后台 |
| 微信小程序 | Taro 3 + React | 微信生态内轻量访问 |
| API 文档 | Swagger UI / Scalar | 对外开放 API 的交互式文档 |
| Hermes UI | 现有 Agent 对话界面 | 无需额外开发，通过 Tool Schema 接入 |

### 2.2 API 层

| 组件 | 技术选型 | 职责 |
|------|----------|------|
| REST API | FastAPI (Python 3.12+) | 核心 CRUD、搜索、数据统计 |
| GraphQL | Strawberry (Python) | 复杂关联查询、前端灵活取数 |
| Webhook | FastAPI Background Tasks | 内容更新通知、Agent 回调 |
| Auth | JWT + OAuth2 (微信/手机号) | 用户认证与权限控制 |

### 2.3 知识引擎层

| 引擎 | 技术选型 | 职责 |
|------|----------|------|
| 全文检索 | Elasticsearch 8 | 关键词搜索、分面筛选、高亮 |
| 语义检索 | Milvus / Qdrant | 向量相似度搜索、RAG 检索增强 |
| 自进化 | Python 定时任务 (Celery) | 内容新鲜度监控、自动采集、AI 摘要 |
| 合规检查 | Python 规则引擎 (Pydantic) | frontmatter 完整性、链接有效性、敏感词过滤 |
| Agent 接口 | MCP / Function Calling Schema | 向 Hermes 暴露 Tool 定义 |
| 知识图谱 | Neo4j Community / NetworkX | 笔记间关系、知识缺口发现 |

### 2.4 数据层

| 存储 | 技术选型 | 职责 |
|------|----------|------|
| 主数据库 | PostgreSQL 16 + pgvector | 用户、笔记、分类、日志、向量检索（轻量场景） |
| 向量数据库 | Milvus (生产) / Qdrant (轻量) | 语义检索专用，支持 1536/768 维向量 |
| 缓存 | Redis 7 | 热点内容缓存、Session、任务队列 |
| 文件存储 | MinIO (自建) / 阿里云 OSS | Markdown 原文件、图片/PDF 附件 |

## 四、与 Hermes Agent 的集成架构

```
┌──────────────────────────────────────────────────────────────┐
│                      用户对话 (Hermes UI)                     │
│  "帮我找一下普者黑的最新数据，写一篇小红书文案"                │
└──────────────────────────┬───────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────┐
│                   Hermes Agent (Orchestrator)                 │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ 1. 意图识别: "搜知识库 + 创作文案"                    │   │
│  │ 2. Tool Call: search_kb("普者黑 最新数据")            │   │
│  │ 3. Tool Call: get_prompt_template("文旅推广")         │   │
│  │ 4. Tool Call: get_material_card("普者黑")            │   │
│  │ 5. Tool Call: check_compliance(content)              │   │
│  │ 6. 综合返回结果 → 生成最终文案                         │   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────┬──────────────────────┬────────────────────────┘
               │ Tool Calls           │ 反馈
               ▼                      ▼
┌──────────────────────────┐  ┌──────────────────────────┐
│    WenShan KB API        │  │   反馈回流                │
│                          │  │                          │
│  /api/tools/search_kb    │  │  Agent 创作内容           │
│  /api/tools/material_card│  │  用户评价数据             │
│  /api/tools/prompt_template│ │  热门查询日志            │
│  /api/tools/compliance   │  │  → 驱动知识库自进化       │
│                          │  │                          │
│  → PostgreSQL            │  │  → 反馈分析 → 内容优化   │
│  → Milvus                │  │  → 缺口发现 → 自动采集   │
│  → 知识引擎               │  │  → 质量评分 → 排序优化   │
└──────────────────────────┘  └──────────────────────────┘
```

**集成要点**：
- Agent 通过 **Function Calling / MCP Protocol** 调用知识库 API，无需修改 Hermes 核心代码
- 每个 Tool 返回标准 JSON，Agent 直接可用
- 反馈回路异步运行，不阻塞对话响应
- 上下文注入由 Agent 决策层控制注入时机和注入量

## 六、核心设计决策

### 6.1 为什么选择 PostgreSQL + pgvector 而非纯 Milvus？

| 场景 | 推荐方案 |
|------|----------|
| 小规模 (< 10 万条)、低并发 | PostgreSQL + pgvector（够用，减少运维） |
| 大规模 (> 10 万条)、高并发 | PostgreSQL + Milvus（专业向量检索，性能更好） |

**建议**：MVP 阶段用 PostgreSQL + pgvector；公测后按需接入 Milvus。

### 6.2 为什么保留 Markdown 原文件？

- Obsidian 知识库继续作为"内容策源地"独立运行
- 迁移不破坏原有结构，`D:\WenShanKB` 保持可读
- 前端可以选择"源码视图"（Markdown 原始格式）或"渲染视图"（富文本）

### 6.3 为什么不直接用 Obsidian Publish？

| | Obsidian Publish | 自建系统 |
|---|---|---|
| 用户管理 | ❌ 无 | ✅ 四级角色 |
| 内容审核 | ❌ 无 | ✅ 审核工作流 |
| API 开放 | ❌ 无 | ✅ REST + GraphQL |
| Agent 集成 | ❌ 不支持 | ✅ Tool Schema |
| 自进化 | ❌ 无 | ✅ 全链路自动化 |
| 成本 | $8/月 | 云服务器 ~￥100/月 |

**结论**：自建系统是面向公共服务的必然选择。
